Databricks vs SageMaker Unified Studio — 기술 검토 관점 비교

2026-04-21 최우연 · Data Engineer / PL PL 관점 회고 · 금융 도메인 데이터 관리 솔루션 구축 사례
Databricks SageMaker Unified Studio Data Governance Unity Catalog Data Platform

한눈에 보기 (핵심 요약)

  • 그룹 통합 데이터 플랫폼 Phase3 데이터 관리 솔루션 구축을 위해 SageMaker Unified Studio와 Databricks를 직접 핸즈온으로 검토했습니다.
  • 초반에는 Databricks로 방향을 잡았으나, 금융권 도메인 데이터가 유입되는 환경 특성과 Databricks 컨트롤 플레인이 Databricks 소유 클라우드 계정에 존재한다는 점 때문에 보안 검토 범위가 크게 확장되었고, 대안으로 SageMaker Unified Studio를 검토하게 되었습니다.
  • 최종 결론은 "데이터 관리 솔루션의 핵심 기능(카탈로그·리니지·품질·권한·감사) 구현에는 Databricks가 더 적합하다" 입니다. 단, 금융권 보안 요건을 충족하기 위한 컨트롤 플레인 격리·계약상의 책임 범위는 별도의 의사결정이 필요합니다.
  • 이 글은 실제 POC를 통해 확인한 기능 차이를, 데이터 거버넌스를 책임지는 PL 관점에서 정리한 기록입니다.

1. 배경 — 왜 두 플랫폼을 동시에 검토했는가

그룹 통합 데이터 플랫폼 Phase3의 목표는 단순한 분석 환경 구축이 아니라, 계열사 전반에 걸친 데이터 자산을 체계적으로 관리할 수 있는 "데이터 관리 솔루션"을 갖추는 것이었습니다. 요구사항은 명확했습니다.

  1. 카탈로그·메타데이터 중앙 관리 — 계열사 데이터가 한 곳에서 탐색/관리되어야 함
  2. 데이터 리니지 자동 수집 — 누가 어떤 데이터를 어떻게 가공했는지 추적 가능해야 함
  3. 데이터 품질 관리 — 품질 기준 설정, 미달 데이터 격리/재적재 가능해야 함
  4. 세분화된 권한 제어와 마스킹 — 금융 도메인 결합 데이터이므로 컬럼 단위 통제 필수
  5. 감사로깅 — 규제 대응을 위한 접근·변경 이력 추적
  6. 외부 포털과의 연동 — 임베딩/API 기반으로 사내 데이터 포털과 결합

초기 내부 리서치 단계에서는 Unity Catalog 기반의 Databricks가 요구사항과 가장 잘 맞는다는 평가가 있었습니다. Lakehouse 아키텍처의 성숙도, 자동 리니지, ABAC, 풍부한 고객 레퍼런스가 이유였습니다.

그러나 금융 도메인이 결합된 데이터가 유입되는 구조라는 점이 의사결정의 축을 흔들었습니다. Databricks는 멀티클라우드를 지원하는 대신 컨트롤 플레인이 Databricks 소유 클라우드 계정에 존재하는 구조입니다. 즉, 메타데이터·작업 제어·사용자 세션과 같은 플랫폼의 "두뇌" 영역이 고객사 VPC 밖에 있습니다. 워크로드가 실행되는 데이터 플레인은 고객사 계정 내에서 동작하지만, 컨트롤 플레인은 Databricks가 운영하는 공간에 있다는 사실은 금융권 정보보안 검토 관점에서 추가적인 위탁평가·CSP 적합성 검토·혁신금융서비스 지정 여부 확인을 필요로 하게 만듭니다.

그래서 "SageMaker Unified Studio로 대체 가능한가"를 동일한 기준으로 다시 평가해야 했고, 그 결과를 공유하기 위해 두 플랫폼을 기능별로 직접 핸즈온 비교하게 된 것이 이번 검토의 배경입니다.

검토 대상 아키텍처 — 두 플랫폼은 "데이터 관리 솔루션 계층"을 어디에 두는가

아래 두 다이어그램은 실제 POC에 사용한 아키텍처 구성도를 단순화해 재구성한 것입니다. 붉은 점선 박스가 "데이터 관리 솔루션"에 해당하는 거버넌스 계층이며, 이 계층의 구성 방식이 플랫폼 선택의 핵심 차이점입니다.

Architecture 1 · SageMaker Unified Studio 기반 구성
포털(콘솔) 자체는 *.sagemaker.<region>.on.aws 형태의 AWS 소유 퍼블릭 엔드포인트로 접근합니다. 거버넌스 계층은 SageMaker Catalog·Glue Data Catalog·Glue Data Quality·Lake Formation·DataZone을 복합 연동해 구성합니다.
Architecture 2 · Databricks 기반 구성
Databricks는 Unity Catalog 단일 계층에서 카탈로그·리니지·품질·권한·감사가 통합 관리됩니다. 다만 컨트롤 플레인은 Databricks Inc. 소유 클라우드 계정에 존재하는 3rd-party SaaS 구조이며, 고객 AWS 계정 내에서는 워크로드가 실행되는 데이터 플레인만 동작합니다.

2. 핸즈온 기반 기능 비교 — 데이터 관리 솔루션 관점

PL로서 관심을 둔 지점은 "어떤 기능이 있는가"가 아니라 "우리가 운영할 때, 이 기능이 추가 구축·관리 부담 없이 당장 쓸 수 있는 수준인가" 였습니다.

2.1 데이터 카탈로그 & 메타데이터 — 통합 관리 계층의 유무

  • Databricks: Unity Catalog 하나로 카탈로그·메타데이터·권한·리니지가 일관된 단일 계층에서 관리됩니다. S3 등 외부 소스와 이기종 DB도 Unity Catalog 아래로 편입시켜 한 번에 탐색할 수 있었습니다.
  • SageMaker Unified Studio: Glue Data Catalog, S3, Redshift, DataZone을 각각 연동해 복합적으로 운영해야 하며, Unity Catalog와 같은 "단일 관리 계층"이 존재하지 않습니다. 도메인 단위 메뉴와 프로젝트 단위 메뉴의 위치·구조가 UI 상 직관적으로 구분되지 않아, 같은 기능을 어디서 접근해야 하는지 조작 동선부터 학습 비용이 발생했습니다.
평가의견 · 데이터 관리 솔루션이 지향해야 할 "단일 진실 원천(single source of truth)"에 가장 가까운 것은 Unity Catalog였습니다. SageMaker Unified Studio는 결국 여러 AWS 서비스를 하나의 UI로 묶어둔 "통합 화면"에 가까웠고, 내부의 자산·권한 모델 자체가 통합되어 있지는 않았습니다.

2.2 데이터 리니지 — "자동 캡처"가 실제로 되는가

  • Databricks: Query Editor·Notebook에서 발생한 작업이 Unity Catalog와 연동되어 자동 리니지로 반영됩니다. Delta Lake 스냅샷 기반으로 이력과 흐름을 한눈에 볼 수 있고, External Lineage(Public Preview)로 외부 시스템 연계까지 확장 가능합니다.
  • SageMaker Unified Studio: 자산 게시·구독 과정의 자동 리니지, Glue Crawler/Visual ETL 리니지는 편리하지만, Athena CTAS, JupyterLab 작업, 외부 ETL 도구에 대한 자동 리니지가 지원되지 않습니다. 해당 구간은 OpenLineage API로 수동 생성해야 합니다.
평가의견 · 데이터 관리 솔루션에서 리니지는 "있느냐"보다 "분석가가 평소 쓰는 도구에서 발생한 변경이 자동으로 잡히느냐"가 본질입니다. CTAS와 노트북은 현업에서 매우 빈번히 쓰이는데, 이 구간을 수동으로 메워야 한다면 사실상 리니지 신뢰성을 보장할 수 없습니다.

2.3 데이터 품질 — 플랫폼이 직접 통제하느냐, 결과만 확인하느냐

  • Databricks: Expectations(DLT)·Data Profiling·Lakehouse Monitoring이 플랫폼 내부에 내장되어 있습니다. 품질 기준을 통과한 데이터만 정상 처리하고, 미달 데이터는 격리 후 재적재까지 파이프라인 안에서 제어할 수 있습니다.
  • SageMaker Unified Studio: 자체 품질 기능은 없고, 외부 서비스인 AWS Glue Data Quality에서 DQDL 규칙을 설정한 뒤, SageMaker Unified Studio는 결과 메트릭만 시각화하여 연계합니다.
평가의견 · "품질 기준을 플랫폼이 강제하느냐"와 "결과를 플랫폼이 보여주느냐"는 전혀 다른 얘기입니다. 전자는 파이프라인 중단/재처리를 자연스럽게 설계할 수 있지만, 후자는 운영 단에서 별도의 후속 워크플로우를 구축해야 합니다.

2.4 권한/보안 — 컬럼 마스킹과 ABAC

  • Databricks: Unity Catalog를 축으로 Row-Level Security, ABAC, Tag Policy 기반 컬럼 마스킹이 일관된 모델로 동작합니다. 금융 도메인 결합 데이터에서 가장 민감한 컬럼 단위 마스킹이 자연스럽게 지원됩니다.
  • SageMaker Unified Studio: 자체 컬럼 마스킹 기능은 없고, Redshift Dynamic Masking 또는 Lake Formation 조합으로 우회 구현해야 합니다. 권한 모델 자체도 RBAC/ABAC보다는 "구독 승인 + 접근 조건 필터"에 가깝고, 스키마·테이블·컬럼 기준의 세밀한 권한 부여는 부자연스럽습니다.
평가의견 · 금융권 결합 데이터 환경에서는 컬럼 마스킹이 기능이 아니라 전제 조건입니다. 우회 구현이 가능하다 해도, 그 책임이 플랫폼 바깥으로 떨어지는 순간 운영 조직의 거버넌스 부담이 급증합니다.

2.5 감사로깅 — 파이프라인을 새로 만들어야 하는가

  • Databricks: system.access.audit 시스템 테이블에 자동 적재됩니다. 별도 파이프라인 없이 SQL로 바로 조회 가능합니다.
  • SageMaker Unified Studio: CloudTrail·CloudWatch Logs에 적재되며, 조회 가능한 형태로 가공하려면 감사로깅 수집·변환·저장 파이프라인을 별도 구축해야 합니다.
평가의견 · 규제 대응 관점에서 감사로깅은 "기본 탑재"여야 합니다. SageMaker Unified Studio는 IAM Identity Center 기반 사용자 단위 추적이 세분화되어 있다는 장점은 있으나, 조회 가능한 로그로 만들기까지의 엔지니어링 비용이 Databricks 대비 확실히 큽니다.

2.6 검색·API·외부 연계

  • 검색: Databricks는 워크스페이스 전체 자산을 아우르는 통합 검색이 강력합니다. 다만 검색 API 미지원이 확인되어, 사내 포털에 임베딩해서 써야 하는 시나리오에서는 제약이 있었습니다. SageMaker Unified Studio는 Asset 검색 API는 있으나 카탈로그 UI 자체의 임베딩이 불가하고, 메뉴 구조가 직관적이지 않았습니다.
  • 노트북 API: Databricks는 REST API 제공. SageMaker Unified Studio는 JupyterLab API 미지원 → 분석 환경에서 구독 승인 데이터를 쓰려면 별도 UI/접근 방식 설계 필요.
  • 대시보드 임베딩: 양쪽 모두 임베딩 URL 제공. Databricks는 툴바 표시 여부 등 옵션 제어의 세밀도가 높습니다.

3. 왜 보안 검토가 확장되었는가 — 컨트롤 플레인 이슈

Databricks의 기능 우위는 명확했음에도 불구하고, 금융권 도메인이 결합되는 데이터 특성상 검토 범위는 단순 기능 비교를 넘어섰습니다. 여기서 중요한 것은 "컨트롤 플레인이 어느 벤더의 운영 영역에 속하느냐"입니다. 엄밀히 말하면 SageMaker Unified Studio 역시 포털 콘솔은 *.sagemaker.<region>.on.aws 형태의 AWS 소유 퍼블릭 엔드포인트로 접근하는 매니지드 서비스이며, 고객 VPC 내부에서 돌아가는 것이 아닙니다. 다만 그 운영 주체가 이미 CSP 계약이 체결된 AWS인지, 별도의 3rd-party 벤더인 Databricks인지에 따라 금융권 관점의 리스크가 달라집니다.

  • 컨트롤 플레인의 운영 주체: Databricks는 사용자 인증·작업 제어·메타데이터 관리가 Databricks Inc.의 클라우드 계정 내에서 이루어지는 3rd-party SaaS 구조입니다. 고객사 입장에서는 기존 CSP(AWS) 외에 새로운 벤더를 신뢰 체인에 추가해야 하고, 이는 위탁평가·계약상 책임 범위의 확장을 의미합니다. 반면 SageMaker Unified Studio는 AWS가 동일한 CSP 계약 범위 내에서 운영하는 매니지드 서비스이므로, 별도 벤더 실사 없이 기존 AWS CSP 적합성 평가 결과를 재사용할 여지가 큽니다.
  • 정보보안 검토 확장: Databricks의 경우 CSP 안정성 평가, 혁신금융서비스 지정 조건(A·B 항목) 충족 여부, 금융보안원 AI 서비스 적합성 확인 등 평가 항목이 기능 검토보다 훨씬 길어지는 상황이었습니다.
  • 망 분리·접근 통제: SageMaker Unified Studio 포털 자체는 퍼블릭 접근이 필요하지만 Private Link 제공이 되고, SageMaker Unified Studio가 사용하는 리소스(Glue, Redshift, S3 등)는 고객 AWS 계정 내에서 VPC/Subnet 단위로 격리할 수 있습니다. Databricks 역시 Private Link 구성이 가능하지만, 이는 Databricks 계정과 고객 계정 간의 별도 네트워크 설계·계약상 합의를 수반합니다.

즉, 이 프로젝트에서 Databricks의 기술적 장점이 모두 유효하더라도, "금융권 도메인 + 신규 3rd-party 벤더 추가"라는 조합은 보안 조직과 수개월 단위의 협의가 필요한 별개의 트랙이었습니다.

4. 결론과 내부 공유 — "기술은 Databricks, 보안은 별도 트랙"

두 플랫폼을 핸즈온 검토한 뒤 내부에 공유한 결론은 다음과 같았습니다.

  1. 데이터 관리 솔루션으로서의 기능 완성도는 Databricks가 명확히 앞선다.
    • Unity Catalog를 중심으로 카탈로그·리니지·품질·권한·마스킹·감사로깅이 일관된 단일 거버넌스 모델로 제공됩니다.
    • SageMaker Unified Studio는 여러 AWS 서비스를 조합해야 하고, 마스킹/감사로깅/리니지 자동화 등 핵심 기능에 우회 구현 또는 추가 개발이 필요합니다.
  2. SageMaker Unified Studio는 "AWS Native를 하나의 화면으로 묶은 편의 레이어"에 가깝다.
    • 도메인/프로젝트 모델로 조직 구조에 맞춘 격리가 편리하고, 포털 자체는 AWS 매니지드 퍼블릭 엔드포인트로 제공되지만 기존 AWS CSP 계약 범위 내에서 운영되므로 보안 검토 관점에서 신규 벤더 실사를 수반하지 않는다는 점이 장점입니다.
    • 다만 2025년 3월 GA로 제품 성숙도와 레퍼런스가 제한적이며, UI 직관성·트러블슈팅 용이성 측면에서 초기 운영 부담이 적지 않았습니다.
  3. 실제 채택 결정은 기술 점수만으로 되지 않는다.
    • "Databricks가 더 적합하다"는 기술 결론은 명확했으나, 컨트롤 플레인이 3rd-party 벤더(Databricks Inc.) 계정에서 운영된다는 구조 자체가 금융권 규제 프레임 안에서는 보안 조직과의 긴 협의를 요구했습니다.
    • PL 관점에서는 "기능 판정"과 "채택 결정"을 분리해서 공유하는 것이 합리적이라 판단했습니다. 기술 검토 결과는 결과대로 공유하되, 보안·컴플라이언스 트랙은 별도의 의사결정 라인을 태웠습니다.

PL로서 얻은 실무 교훈

  • 도입 타당성 검토는 "기능 표"가 아니라 "운영 시나리오"로 해야 합니다. 리니지가 지원되느냐가 아니라, 분석가가 실제로 쓰는 도구(Query Editor, JupyterLab)에서의 변경이 자동으로 잡히느냐가 본질입니다.
  • 금융 도메인에서는 "컨트롤 플레인이 어느 벤더의 운영 영역에 속하는가"가 플랫폼 선택의 1급 변수가 됩니다. 두 플랫폼 모두 포털은 퍼블릭 엔드포인트로 제공되지만, 그 운영 주체가 기존 CSP 계약 범위 안에 있는지 아닌지가 위탁평가 부담을 결정합니다. 멀티클라우드 장점은 일반 산업에서는 매력이지만, 이 업권에서는 신뢰 체인에 새 벤더를 추가하는 비용으로 작동합니다.
  • 핸즈온 없이 벤더 자료로 판정하지 않도록 두 플랫폼 모두 실제 POC를 태웠고, 이것이 내부 공유 시 설득력을 크게 높여 주었습니다. 특히 SageMaker Unified Studio의 "자동 리니지 지원"이 실제로는 JupyterLab/CTAS에서 동작하지 않는다는 점은 POC 없이는 확인하기 어려운 항목이었습니다.
  • "결론 한 줄 + 비교표 한 장"을 같은 페이지에 함께 보여주세요. "Databricks가 더 적합합니다"라는 한 줄 메시지 바로 아래에, 기능별로 어디서 얼마나 차이가 나는지 정리한 비교표를 붙여두면 됩니다. 그래야 보고를 받는 사람이 왜 그런 결론이 나왔는지 스스로 납득할 수 있고, 같은 논의를 다시 반복하지 않아도 됩니다.

5. 맺음말

이번 검토는 단순히 "어떤 제품이 더 좋은가"를 고르는 작업이 아니었습니다. 데이터 관리 솔루션을 누가·어디서·어떻게 운영할 것인지, 그리고 금융 도메인이 결합되는 환경에서 컨트롤 플레인이라는 추상적 개념이 얼마나 구체적인 리스크로 전환되는지를 팀과 함께 확인해 가는 과정이었습니다.

두 플랫폼 중 어느 쪽을 최종 채택하든, "우리 조직이 감당할 수 있는 거버넌스 부담의 총량"이 기준이 되어야 한다는 점이 이번 검토의 가장 큰 수확이었습니다. 기술 우위가 명확해도 보안·컴플라이언스 리스크가 운영 조직의 역량을 초과한다면 채택할 수 없고, 반대로 보안 적합성이 높아도 기능 공백을 우회 구현으로 메우는 비용이 장기적으로 조직을 소모시킨다면 그 역시 현명한 선택이 아닙니다.

이 글이 비슷한 지점을 고민하는 데이터 플랫폼 담당자·PL분들께 한 조각의 참고가 되기를 바랍니다.
