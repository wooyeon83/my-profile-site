하루 9GB 로그 배치를 3분 40초로 줄인 Flume + Spark Streaming 튜닝 이야기

2026-04-20 최우연 · Data Engineer
Spark Streaming DStream Flume Scala Performance

1. 들어가며 — 하루 9GB 로그가 10분 넘게 밀리던 이야기

당시 운영하던 곳은 대형 통신사급 로그 플랫폼이었다. 하루 대략 9GB 분량의 이벤트 로그를 받아 실시간 집계 결과를 모니터링 대시보드에 띄워주는 파이프라인이었는데, 파이프라인 자체는 멎지 않고 잘 돌아갔다. 문제는 항상 10~15분쯤 늦게 보인다는 것이었다. 운영자 입장에서는 "지금 장애인가?"를 판단해야 하는데, 차트는 10분 전 상황을 보여주니 실시간 모니터링이라고 부르기 민망했다.

이 글은 그 파이프라인을 내려받아 뜯어보며, 결국 한 배치당 3분 40초 빨라지는 이유를 "아, 이래서 빨라지는구나" 하고 느낄 수 있도록 정리한 글이다. Kafka가 아닌 Flume + Spark Streaming(DStream) 조합이라, 요즘 자료에는 잘 안 나오는 포인트도 함께 적어뒀다.

2. 시스템 구성 — Flume이 Spark Streaming으로 직접 쏟아붓는 구조

요즘은 로그 수집이라 하면 보통 Kafka를 떠올리지만, 이 시스템이 설계된 시점에는 Apache Flume이 표준이었다. 간단히 설명하면, Flume은 세 부품으로 이루어진다.

  • Source: 로그를 끌어오는 입력부. tail 기반, spooling directory 기반 등 다양하다.
  • Channel: 받은 이벤트를 Sink로 넘기기 전 잠시 담아두는 버퍼. 비유하자면 버스 터미널의 대기실이다.
  • Sink: 최종 목적지로 이벤트를 내보내는 출력부.

그리고 이 파이프라인의 특이점은, Flume의 Sink가 Spark Streaming이었다는 점이다. Kafka 같은 중간 브로커가 없었다. 즉 Spark Streaming이 FlumeUtils를 통해 직접 Flume의 이벤트를 받아 DStream으로 소비하는 구조였다. 

  [Edge Servers / Log Files]
            |
            v
   +------------------+
   |   Flume Source   |   (spooling / exec / tail)
   +------------------+
            |
            v
   +------------------+
   |  Flume Channel   |   Memory Channel
   |  (대기실)         |   capacity / transactionCapacity
   +------------------+
            |
            v
   +------------------------------------+
   | Flume Sink = Spark Streaming       |
   |   FlumeUtils.createPollingStream   |   <-- Pull 방식
   |   또는 createStream (Push 방식)     |
   |                                    |
   |   DStream -> reduceByKey / window  |
   |   foreachRDD -> HDFS / HBase       |
   +------------------------------------+
            |
     +------+------+
     v             v
  +--------+   +--------+
  |  HDFS  |   | HBase  |
  +--------+   +--------+

Spark 쪽에서는 대략 이렇게 스트림을 열었다. DStream API를 쓰는 Java 코드다. 

// Pull 방식(권장) — Spark executor가 Flume의 Sink에서 이벤트를 당겨온다
List<InetSocketAddress> addresses = Arrays.asList(
    new InetSocketAddress("flume-agent-01", 41414),
    new InetSocketAddress("flume-agent-02", 41414));

JavaReceiverInputDStream<SparkFlumeEvent> flumeStream =
    FlumeUtils.createPollingStream(
        jssc,
        addresses,
        StorageLevel.MEMORY_AND_DISK_SER_2());

flumeStream
    .map(event -> parse(event.event().getBody().array()))
    .filter(LogRecord::isValid)
    .mapToPair(e -> new Tuple2<>(e.getKey(), 1L))
    .reduceByKey((a, b) -> a + b)
    .foreachRDD(rdd -> writeToHBase(rdd));

처음엔 이 구조가 단순해서 좋다고 생각했다. 하지만 지연 원인을 쫓다 보니, 단순한 만큼 모든 부담이 한쪽으로 쏠린다는 사실을 깨달았다.

3. 병목은 어디에 있었나 — 신호를 먼저 읽는다

튜닝 이야기에서 가장 피하고 싶은 건 "이 설정을 이렇게 바꿨다"만 따라하는 일이다. 같은 파라미터도 환경이 다르면 역효과를 낸다. 그래서 손대기 전에, 어떤 신호를 보고 판단했는지부터 먼저 적어둔다.

3.1 Spark Streaming UI — 어디서 열고, 무엇을 보나

Spark 애플리케이션이 떠 있으면 드라이버 머신에서 자동으로 웹 UI가 열린다. 주소는 보통 http://<driver-host>:4040이고, StreamingContext가 있으면 상단에 "Streaming" 탭이 자동으로 생긴다. 일단 그 탭만 열면 판단에 필요한 숫자는 대부분 한눈에 들어온다. 

Streaming Statistics (예시 화면 수치)
------------------------------------------------
Batch Duration    : 30 s
Input Rate        : 평균 12,000 rec/s (피크 35,000)
Scheduling Delay  : 평균  45 s   ← 배치 간격보다 크면 밀리는 중
Processing Time   : 평균  48 s   ← 한 배치 처리에 드는 시간
Total Delay       : 평균  93 s   ← 사용자가 체감하는 전체 지연
------------------------------------------------
  • Scheduling Delay > Batch Interval이면 큐가 밀리고 있다는 뜻. 줄이 점점 길어지는 상태이고, 가장 먼저 볼 신호다.
  • Processing Time이 튀는지: 평균은 괜찮은데 특정 배치만 오래 걸리면 보통 skew나 GC 문제다. 해당 배치를 클릭하면 Stage 단위로 어디에서 시간을 썼는지 보인다.
  • Input Rate의 모양: 평평한지, 뾰족한지에 따라 백프레셔가 필요한지가 갈린다.

4040 포트는 드라이버가 살아 있는 동안만 열려 있다. 애플리케이션이 끝난 뒤에도 지표를 되짚어보고 싶다면 Spark History Server(기본 18080 포트)에서 완료된 배치의 스테이지 타임라인을 다시 볼 수 있다.

3.2 Flume JMX 메트릭 — 켜는 법부터 보는 법까지

Flume agent도 JMX로 내부 상태를 내보낼 수 있는데, 기본 설치 상태에서는 꺼져 있는 경우가 많다. 먼저 conf/flume-env.sh에 옵션을 추가해 agent를 재시작한다. 

# conf/flume-env.sh — JMX 활성화
export JAVA_OPTS="$JAVA_OPTS \
  -Dcom.sun.management.jmxremote \
  -Dcom.sun.management.jmxremote.port=41415 \
  -Dcom.sun.management.jmxremote.authenticate=false \
  -Dcom.sun.management.jmxremote.ssl=false"

이후에는 JConsole이나 jmxterm 같은 CLI 도구로 접속해서 Channel·Source·Sink의 MBean을 조회한다. 운영에서는 Prometheus JMX Exporter를 붙여 Grafana로 지속 모니터링하는 게 편하지만, 급하게 상태만 보고 싶을 땐 jmxterm이 가장 빠르다. 

# jmxterm으로 Channel 상태 조회 (예시)
$ java -jar jmxterm.jar --url flume-agent-01:41415

$> bean org.apache.flume.channel:type=memCh
$> get ChannelFillPercentage EventPutSuccessCount EventTakeSuccessCount
ChannelFillPercentage   = 82.4          # 채널이 82% 차 있다
EventPutSuccessCount    = 14,823,910    # Source가 넣은 누적 건수
EventTakeSuccessCount   = 14,812,477    # Sink가 꺼낸 누적 건수

위 예시에서는 두 카운터의 차이가 약 1만 건이고 Fill이 82%다. 들어오는 속도가 나가는 속도보다 살짝 빠르다는 뜻이며, 이 상태가 이어지면 결국 100%로 포화된다. 이때의 Channel이 바로 상류 병목 신호다. 주로 보는 지표는 다음과 같다.

  • ChannelFillPercentage: 채널이 얼마나 차 있는지. 피크에 90%+가 잦으면 capacity부터 의심.
  • EventPutSuccessCount / EventTakeSuccessCount: 넣은 건수와 꺼낸 건수. 두 값의 증가 기울기를 비교하면 어느 쪽이 느린지 판단이 된다.
  • ChannelSize: 현재 채널에 남아 있는 이벤트 수. Fill 값의 원재료다.

개선 전 상태는 두 가지가 동시에 나타났다. Flume Channel Fill이 자주 포화에 가까웠고, Spark UI에서는 Scheduling Delay가 계단식으로 쌓였다. 병목은 한 군데가 아니라 Flume Channel → Spark 처리 시간 → 피크 흡수 능력의 세 지점에 걸쳐 있었다. 그래서 튜닝 축도 셋으로 나눠 접근했다.

4. 튜닝 1 · Flume Channel 용량부터 늘렸다

증상

피크 시간대에 Flume Channel Fill이 95%를 넘으며 Source가 이벤트를 밀어넣지 못해 정체. 상류 로그 파일은 계속 쌓이는데 파이프라인은 숨이 차 있었다.

원인

Memory Channel의 capacity(대기실에 담을 수 있는 최대 이벤트 수)가 기본값(1,000) 근처로 설정돼 있었고, Sink가 한 번에 꺼내가는 transactionCapacity도 100 수준이라 트랜잭션 오버헤드가 컸다.

처방

대기실 자체를 넓히고, 버스 한 대가 한 번에 태우는 인원(배치 크기)도 늘렸다.

결과

피크 시 Channel Fill이 40~60% 범위에서 안정. Source가 블로킹되는 빈도가 확연히 줄어 지연 누적의 시작점 자체를 제거.

실제로 만진 파라미터는 크게 세 개였다. Flume 설정 파일 기준으로 아래와 같은 형태다. 

# 예시 flume-agent.conf (Memory Channel + Spark Streaming sink)

agent.channels.memCh.type = memory
agent.channels.memCh.capacity             = 200000   # (이전: 1000)
agent.channels.memCh.transactionCapacity  = 5000     # (이전: 100)
agent.channels.memCh.keep-alive           = 30       # (이전: 3)

# Source 측 배치 크기도 함께 맞춰줘야 의미가 있다
agent.sources.src1.batchSize              = 2000

# Sink(=Spark Streaming이 polling으로 가져감) 측 배치 크기도 정렬
agent.sinks.sparkSink.batchSize           = 2000

각 파라미터가 왜 중요한지 한 줄씩 풀어보면 이렇다.

  • capacity: 대기실의 총 좌석 수. 이게 작으면 조금만 피크가 와도 Source가 멈춘다. 단, 메모리 채널이므로 JVM 힙을 고려해 무작정 올리면 OOM이 난다. 우리 환경에서는 힙을 함께 올려 20만 건 정도까지 잡았다.
  • transactionCapacity: Sink가 한 트랜잭션에 꺼내갈 수 있는 최대 이벤트 수. 너무 작으면 Sink가 쉴 새 없이 짧은 트랜잭션을 반복하고, 너무 크면 트랜잭션이 실패했을 때 재처리 비용이 커진다. capacity의 2~5% 수준이 안정적이었다.
  • keep-alive: Channel이 가득 찼거나 비었을 때 얼마나 기다릴지. 짧으면 바쁜 시간대에 예외가 자주 난다. 3초 → 30초로 늘리자 로그의 "Channel full" 예외가 사라졌다.
꿀팁. transactionCapacity는 Source의 batchSize, Sink의 batchSize와 반드시 같거나 크게 맞춰야 한다. 한쪽만 키우면 "트랜잭션 용량이 batch size보다 작다"는 예외로 Flume agent 자체가 뜨지 않는 사례를 여러 번 봤다.

5. 튜닝 2 · Spark Streaming 파티셔닝과 Shuffle

증상

Stage 타임라인에서 상위 스테이지 몇 개가 전체 시간의 70%를 먹고 있었고, 그 안에서도 소수의 태스크가 나머지를 기다리게 만드는 skew가 뚜렷했다.

원인

(1) Flume receiver 개수가 executor core 수에 비해 적어 초기 병렬성이 안 나왔고, (2) 코드 일부에 groupByKey가 있어 셔플 데이터가 과도, (3) 일부 핫 키 때문에 한 태스크에 데이터가 쏠림.

처방

receiver 수 / parallelism / 집계 연산 / salting을 차례로 손봤다.

결과

상위 스테이지의 Shuffle Write가 체감될 만큼 줄고, 태스크별 처리 시간이 한쪽으로 쏠리지 않고 고르게 퍼졌다. 가끔 튀던 "느린 배치"까지 눈에 띄게 빨라졌다.

5.1 Receiver 개수와 Executor core 정렬

DStream + Flume 구조에서는 createPollingStream에 넘기는 Flume agent 주소가 여러 개면 executor 여러 개가 나눠서 당겨온다. 초기 구성은 주소가 하나뿐이라 한 executor가 모든 수신을 혼자 처리하고 있었다. 대기실이 넓어져도 문을 통과시키는 직원이 한 명이니 줄이 서는 셈이다.

Flume agent 두 대를 병렬로 두고 Spark 쪽에서 두 주소를 동시에 구독하게 바꿨다. 동시에 executor 개수와 core 수도 정렬했다. 경험상 receiver 수는 executor 수의 약수로 맞추는 편이 관리가 쉬웠다.

5.2 Shuffle 병렬성 — 태스크당 처리 시간으로 판단

sparkConf.set("spark.default.parallelism",    "200");    // 셔플/리파티션 기본값
sparkConf.set("spark.sql.shuffle.partitions", "200");    // DF 경유 구간 대비
sparkConf.set("spark.locality.wait",          "500ms");  // 로컬리티 집착 완화
sparkConf.set("spark.serializer",
    "org.apache.spark.serializer.KryoSerializer");

숫자 자체는 환경에 따라 다르다. 우리는 태스크당 평균 처리 시간이 수백 ms 수준이 되도록 partition 수를 잡았다. 너무 잘게 쪼개면 태스크 스케줄링 오버헤드가 전체 시간을 잡아먹고, 너무 크게 잡으면 병렬성이 안 나온다. 평균값보다 태스크 처리 시간의 분포를 보는 게 더 유용했다.

5.3 reduceByKey로 바꾸고, 핫 키만 salting

코드에 groupByKey().mapValues(_.sum) 같은 구간이 있었다. 이 패턴은 셔플 전에 지역 집계가 일어나지 않아 네트워크로 원본 값들을 전부 보낸다. reduceByKey(_ + _)로 바꾸면 각 파티션에서 map-side combine이 먼저 일어나 셔플 데이터 자체가 몇 배로 줄어든다. 이 한 줄 교체만으로 해당 스테이지의 Shuffle Write가 눈에 띄게 감소했다.

그래도 남는 skew가 있었다. 상위 몇 개 키에 데이터가 쏠리는 것. 그 키들만 골라 접미사 0~k를 랜덤으로 붙여 1차 집계 → 접미사 제거 → 2차 집계의 두 단계로 분산시켰다 (소위 salting). 모든 키에 적용하면 단계만 늘어 손해라, 실제로 쏠리는 키만 선별적으로 적용하는 게 핵심이었다.

6. 튜닝 3 · 배치 간격과 백프레셔

증상

평상시엔 Scheduling Delay가 낮은데, 피크가 한번 오면 그 영향이 10~20분간 이어지며 해소가 느렸다. 재시작 직후 첫 배치가 유독 크게 튀는 문제도 있었다.

원인

Batch Interval이 Processing Time에 너무 바짝 붙어 여유가 없었고, 백프레셔 관련 설정이 꺼져 있어 트래픽 스파이크가 그대로 Processing Time으로 전이됐다.

처방

Interval을 약간 늘려 여유를 두고, backpressure + 초당 레코드 상한을 세트로 걸었다.

결과

피크에서도 Scheduling Delay가 Batch Interval 이내로 안정. 재시작 직후 첫 배치 스파이크도 사라졌다.

6.1 Batch Interval은 "처리 시간의 60~70% 아래가 평상시" 기준

Batch Interval은 Spark Streaming이 "얼마마다 한 번씩 배치를 만들지" 결정하는 값이다. 처음 배울 때는 "짧게 잡을수록 실시간에 가깝다"고 생각하기 쉬운데, 현장에서는 오히려 반대 현상이 자주 일어난다. Interval이 Processing Time보다 아주 조금만 크면, 작은 스파이크도 바로 Scheduling Delay로 번진다.

우리 환경에서는 평상시 Processing Time이 Batch Interval의 60~70% 정도에 머물도록 잡았다. 그래야 트래픽이 살짝 튀어도 바로 지연으로 이어지지 않는다. Interval을 억지로 줄이려 하기보다는, Interval을 살짝 늘려 여유를 두고 앞 단계(파티셔닝, Flume Channel)에서 Processing Time을 떨어뜨리는 쪽이 훨씬 안정적이었다.

6.2 Backpressure — 입국장 출입구를 상황 따라 조절

Backpressure는 Spark Streaming이 "지금 내가 처리 속도를 못 따라가고 있으니, 앞 단계에서 내보내는 속도를 알아서 조절해다오"라고 요청하는 기능이다. 입국장에서 심사가 밀리면 게이트를 좁혀 들어오는 인원을 줄이는 그림과 같다. 

sparkConf.set("spark.streaming.backpressure.enabled",     "true");
sparkConf.set("spark.streaming.backpressure.initialRate", "20000");
sparkConf.set("spark.streaming.receiver.maxRate",         "8000");
sparkConf.set("spark.streaming.backpressure.pid.minRate", "1000");

여기서 중요한 건 spark.streaming.receiver.maxRate다. receiver 하나가 초당 받아들일 수 있는 최대 레코드 수의 하드 캡이다. 이걸 안 걸어두고 백프레셔만 켜두면, 특히 재시작 직후에 첫 배치가 수 분 단위로 튀는 현상이 생긴다. 그 사이 Flume Channel에 쌓였던 이벤트를 한 번에 빨아들이려 하기 때문이다. 두 설정은 반드시 세트로 움직여야 했다. 운영 중에 몸으로 배운 규칙이다.

7. 부수적으로 효과가 있던 것들

핵심 튜닝 축 셋이 가장 큰 개선을 가져왔고, 나머지 설정들은 그 위에 얹는 마감재 정도였다. 짧게만 적는다.

  • Kryo 직렬화: 자바 기본 직렬화에서 Kryo로 바꾸고, 자주 쓰는 케이스 클래스를 registerKryoClasses로 등록했다. 셔플이 큰 스테이지에서 직렬화 비용이 체감될 정도로 줄었다.
  • Executor memory overhead 비율: 피크 시 OOM-killer로 executor가 사라지는 일이 간헐적으로 있었다. executor memory를 늘리기 전에 overhead 비율부터 조정하는 게 먼저였다.
  • GC 튜닝: 일부 executor의 GC Time이 태스크 시간의 15%를 넘고 있었다. G1GC로 바꾸고 힙 크기를 조절하는 정도에서 그쳤지만, 꼬리 지연이 줄었다.

8. 결과 — 10~15분에서 3분 40초로

지표BeforeAfter개선
배치 처리 지연10~15분3분 40초약 3~4배
Flume Channel Fill피크 95%+40~60% 안정상류 블로킹 해소

피크 후 회복 시간10~20분1~2분백프레셔 효과

작업을 마치며 느낀 점 세 가지

  1. 상류부터 보자. Spark 튜닝 이야기는 많지만, Flume 같은 상류가 막혀 있을 땐 Spark만 아무리 만져봐야 결과가 잘 나오지 않았다. 지연이 어디에서 시작되는지부터 지표로 확인하는 습관이 생겼다.
  2. 설정은 혼자 움직이지 않는다. capacity와 transactionCapacity와 batchSize, backpressure와 receiver.maxRate, parallelism과 executor core는 언제나 세트로 함께 봐야 했다. 한쪽만 바꾸면 다른 쪽이 발목을 잡는 경우를 여러 번 겪었다.
  3. 평균보다 "가장 느린 배치"를 보자. 평균 처리 시간이 Batch Interval 안에 들어오더라도, 가끔 튀는 느린 배치가 쌓이면 결국 밀린다. 느린 쪽을 기준으로 여유를 잡아두니 운영이 한결 조용해졌다.

9. 마치며 — 지금이라면 Structured Streaming + Kafka

이 경험은 DStream API 시절, Flume이 현역이던 때의 기록이다. 지금 같은 과제를 새로 설계한다면 나는 Flume 대신 Kafka를, DStream 대신 Structured Streaming과 이벤트 타임 워터마크, 상태 저장소 기반 집계를 선택할 것이다.

그렇지만 여기서 다룬 원칙들이 낡았다고 생각하지는 않는다. 상류 버퍼의 용량, 병렬성, 배치 간격, 백프레셔, 느린 배치 관리. 엔진이 바뀌어도 이 다섯 가지를 함께 움직여야 실시간 파이프라인이 조용히 돈다는 감각은 그대로다. 결국 실시간 처리의 성능은 마법 같은 설정 하나가 아니라, 어떤 지표를 어떤 순서로 볼지를 아는 데서 나온다.

본 글에 사용된 수치와 아키텍처는 NDA 준수를 위해 일반화되었습니다. 특정 고객사, 도메인, TPS, 스키마, 실제 코드는 포함하지 않았으며, 공개된 이력서에 기재된 개선 수치(약 10~15분 → 3분 40초) 범위 안에서만 서술했습니다.
