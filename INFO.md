## 채용 담당자 관점 체크리스트

채용 담당자가 포트폴리오를 볼 때 평균 30초 ~ 2분 이내 아래 질문에 답이 보여야 합니다.

- [ ] 이 사람의 **핵심 직무/연차**가 3초 안에 보이는가? (Data Engineer / 18년+)
- [ ] **대표 성과 1~3줄**이 한눈에 들어오는가?
- [ ] **이력서 PDF**를 바로 다운로드할 수 있는가?
- [ ] **연락 수단**(이메일·LinkedIn 등)이 명확한가?
- [ ] **대표 프로젝트의 정량적 임팩트**가 드러나는가?
- [ ] **실제 코드/결과물**(GitHub, 블로그, 아티팩트)을 확인할 수 있는가?
- [ ] **모바일 환경**에서도 정상 작동하는가?

## 우선순위 [높음] — 서류전형 이전에 반드시 추가

### 1. 이력서 PDF 다운로드 버튼
- Hero / Contact 섹션에 **"이력서 다운로드"** CTA 배치
- 파일명 예시: `최우연_DataEngineer_Resume_2026.pdf`
- 채용 담당자가 별도 요청 없이 바로 열람 가능해야 함

### 2. 프로젝트별 정량적 성과(Impact Metrics) 강화
현재: "처리 시간 단축", "성능 개선" 수준의 서술
개선: **수치 중심**으로 재구성
- 예) **"9GB 로그 분석 시간을 3분 40초까지 최적화 (기존 대비 XX% ↓)"**
- 예) **"수십 TB 규모 데이터 처리 파이프라인 안정화로 SLA 99.9% 달성"**
- 예) **"Redshift 쿼리 부하 제거로 대시보드 로딩 시간 N초 → N초 단축"**

각 프로젝트 카드에 아래 3요소를 포함 권장
1. **Problem** — 어떤 기술·비즈니스 문제였는지
2. **Solution** — 어떤 기술적 결정으로 해결했는지
3. **Impact** — 수치로 보이는 결과

### 3. GitHub 프로필 링크 추가
- 현재: LinkedIn만 노출됨
- 실제 코드 스타일을 확인할 채용 담당자가 많음
- README가 잘 정리된 대표 레포 3~5개를 핀 고정
- 최소 **토이 프로젝트 1~2개**라도 공개 권장
  (Airflow DAG 샘플, PySpark 튜닝 예제, Terraform 모듈 등)

### 4. 기술 블로그 / Medium / Velog 링크
- 경력 연차 대비 **글 1~2건**이라도 노출되면 기술 사고력 증명에 큰 도움
- 추천 주제 예시
  - PySpark 성능 튜닝 사례 (메모리/파티셔닝)
  - Airflow 대규모 DAG 운영 노하우
  - Databricks vs SageMaker Unified Studio 기술 비교 (최근 업무와 연결)
  - AWS MWAA 운영 팁

### 5. 핵심 성과 하이라이트 카드 (Career Highlights)
Hero 아래 또는 About 위에 **"3대 핵심 성과"** 시각적 배치
- 수십 TB 실시간 데이터 처리 시스템 구축
- 그룹사 통합 데이터 포털 PL 수행 (2024~)
- AWS ML Specialty + ADsP 취득
- AI 빅데이터학 석사 (국민대, 2026.02 예정)

## 우선순위 [중간] — 여유가 있다면 추가

### 6. 아키텍처 다이어그램
- 대표 프로젝트 1~2개에 **데이터 흐름도**(SVG/PNG) 첨부
- 추천 후보: 그룹 통합 데이터 포털, KAIST 데이터 레이크
- 기술 선택 근거와 trade-off를 1~2줄 설명하면 설계 역량이 잘 드러남

### 7. 프로젝트 상세 페이지 (Case Study)
현재: 카드형 요약만 제공
개선: 카드 클릭 시 모달 또는 `/projects/{slug}` 상세 페이지로 확장
- 배경 & 문제 정의
- 기술적 제약 / 의사결정
- 구현 포인트 (2~3개 핵심)
- 결과 & 배운 점 / 다음 개선안

### 8. 리더십 & 협업 경험 섹션
이력서상 **PL 수행** 경험이 눈에 띄지 않음 → 별도 강조 필요
- HR 채용 예측 모델 (PL)
- 그룹 통합 데이터 포털 (PL)
- KTDS, 관계사, 교보정보통신 등 **다자간 협업** 경험

### 9. 기술 스택 숙련도 레벨 표시
단순 나열 → 3단계(Expert / Advanced / Familiar)로 구분
- 예) Expert: Java, Python, AWS, Spark, Airflow
- 예) Advanced: Kotlin, Tableau, Databricks, BigQuery
- 예) Familiar: Terraform, Elasticsearch, GBM

## 보안 / 개인정보 관련 주의사항

### 반드시 확인할 항목
1. **휴대폰 번호 공개 여부**
   - 현재 Contact 섹션에 `+82-10-5656-5794` 노출 중
   - 공개 포트폴리오 URL인 경우 이메일·LinkedIn만 노출하는 것이 안전
   - 지원 회사에만 전달한다면 유지 가능

2. **프로젝트 기밀 / NDA**
   - 교보생명, SK브로드밴드, 삼성전자 등 고객사 관련 프로젝트의 상세 구조·수치는 NDA 검토 필요
   - "수십 TB", "3분 40초" 등은 이미 이력서에 공개된 수준이면 OK

3. **이력서와 사이트 내용 동기화**
   - 두 문서의 경력·프로젝트·수치가 어긋나면 신뢰도 하락
   - 업데이트 시 양쪽 모두 함께 반영

## 기술적 개선 제안

### 배포 & URL
- 현재 로컬 파일 → GitHub Pages 또는 Vercel 배포 권장
- 추천 URL 형태: `wooyeon.dev` / `wooyeon-choi.vercel.app` / `github.io/wooyeon-choi`
- **HTTPS 필수** (브라우저 경고 방지)

### 빌드 & 유지보수
- 현재 단일 `index.html` + CDN Tailwind → 빠른 프로토타입엔 OK
- 장기 유지보수·섹션 추가를 고려하면 프로젝트 지시사항(`CLAUDE.md`)에 맞춰 **React + TypeScript + Tailwind**로 마이그레이션 고려
- 프로젝트 데이터는 JSON/MDX로 분리하면 추가·수정이 편리

### 분석
- Google Analytics 4 또는 Plausible 연동
- 어떤 회사에서 얼마나 체류하는지 파악 가능 (지원 전략에 활용)

## 참고 — 훌륭한 데이터 엔지니어 포트폴리오 공통점

1. **수치로 말한다** — "개선했습니다" ✗ / "쿼리 시간 N초 → N초 (N% ↓)" ✓
2. **의사결정 과정이 보인다** — 단순 기술 나열이 아닌 "왜 이 선택을 했는가"
3. **최신성** — 최근 6개월 이내 업데이트 흔적 (블로그 글·GitHub 커밋)
4. **스토리** — 18년의 커리어를 한 줄 스토리로 정리할 수 있어야 함
   ("임베디드 → 백엔드 → 데이터 플랫폼 → AI/ML"로 이어지는 **진화 서사** 강조 가능)
5. **신뢰 신호** — 자격증, 학위, 추천사, 수상 이력
