# 시니어 데이터 엔지니어를 위한 GitHub 약점 보완 전략

> 작성일: 2026-04-18
> 대상: 실무 코드가 모두 기업 NDA로 공개 불가한 시니어 데이터 엔지니어
> 현 상황: GitHub에 대학원 과제 코드만 존재

---

## 마인드셋 전환 (가장 중요)

채용 담당자는 시니어 엔지니어에게 **"얼마나 많은 코드를 공개했는가"**를 보지 않습니다.
오히려 **"어떤 문제를 어떤 이유로 어떻게 풀었는가"**를 평가합니다.

따라서 GitHub 코드가 적더라도 괜찮습니다 — **대체 수단**을 제대로 갖추면 됩니다.

> **핵심 원칙**: 시니어일수록 `how to code` 보다 `how to think`가 평가 대상입니다.

---

## 권장 보완 전략 (임팩트 × 난이도 기준)

### 최우선 순위 (시작 1주 내)

#### 1. GitHub 프로필 README 상단에 "컨텍스트 배너" 추가

채용 담당자가 오해하지 않도록 **먼저 설명**하는 것이 핵심입니다.

예시 구조:
```markdown
# 안녕하세요, 데이터 엔지니어 최우연입니다 👋

18년차 Data Engineer. 대부분의 실무 코드는 기업 NDA로 공개할 수 없습니다.
대신 **기술 블로그**, **아키텍처 문서**, **포트폴리오 사이트**로 실무 경험을 공유합니다.

### 🔗 이곳에서 저를 더 잘 볼 수 있습니다
- 📝 기술 블로그: [링크]
- 💼 LinkedIn: [링크]
- 🌐 포트폴리오: [링크]

### 📂 이 GitHub에는
- 국민대학교 대학원 AI 빅데이터학과 과제 (2024~2026)
- 공개 가능한 샘플·템플릿 프로젝트
- 개인 학습 토이 프로젝트
```

#### 2. 기술 블로그 글 1~2편 작성 (가장 효과적)

회사 코드 없이도 **아키텍처·의사결정·트러블슈팅**은 공유 가능합니다.

**이력서 기반 추천 주제 (NDA 안전)**
- "9GB 로그 실시간 분석을 3분 40초로 줄인 Spark Streaming 튜닝 경험"
- "Databricks vs SageMaker Unified Studio, 기술 검토 관점 비교"
- "AWS MWAA로 그룹 단위 Airflow 운영하며 배운 것들"
- "GBM + Permutation Importance로 HR 채용 예측 모델 설계"
- "Terraform으로 데이터 레이크 인프라 자동화한 경험"

**NDA 안전 서술 원칙**
- 회사명은 "대형 금융사", "통신사", "커머스 플랫폼" 수준으로 일반화
- 고객 데이터·테이블 스키마·비즈니스 로직 구체 수치 제외
- 대신 "아키텍처 결정 근거"·"트레이드오프"·"성능 개선 퍼센트"는 공유 가능

**추천 플랫폼**
- Medium, Velog, 개인 블로그 (GitHub Pages)
- 회사 기술 블로그에 기고 가능하면 이상적

#### 3. 대학원 과제 레포 README 정리

이미 있는 코드를 **잘 포장**만 해도 인상이 크게 달라집니다.

**각 레포 README 체크리스트**
- [ ] 프로젝트 배경 (왜 이 문제를 풀었는가)
- [ ] 사용 기술 스택
- [ ] 실행 방법 (Quick Start)
- [ ] 결과 스크린샷·차트·지표
- [ ] 배운 점·한계·다음 개선안

국민대 AI 빅데이터학과 과제는 **최신 기술 학습 의지**의 증거가 됩니다 — 숨길 것이 아니라 **자산**으로 활용하세요.

---

### 2순위 (1~2개월 내)

#### 4. 샘플·템플릿 레포 2~3개 공개

실무 패턴을 **일반화한 버전**으로 재현 — NDA 위반이 아닙니다.

**추천 레포 아이디어**
- `airflow-etl-template` — 파라미터화된 ETL DAG 보일러플레이트
- `pyspark-performance-tips` — 파티셔닝·브로드캐스트 조인 벤치마크
- `terraform-aws-datalake` — S3 + Glue + Athena 기본 구성
- `spark-streaming-examples` — Flume → Spark Streaming 로그 처리 예제
- `bi-tableau-data-mart` — Tableau 최적화된 데이터 마트 설계 예제

**각 레포 README에 명시할 문구**
> "실무에서 반복적으로 마주하는 패턴을 공개 가능한 형태로 정리한 템플릿입니다. 실제 구현 세부는 기업 환경에 최적화되어 있으나, 구조·철학은 이 레포의 것과 유사합니다."

#### 5. 공개 데이터셋 기반 End-to-End 토이 프로젝트 1개

시간이 많이 드므로 **하나만 제대로** 만드는 것을 추천합니다.

**예시 구성**
- 데이터: 공공데이터포털, 서울시 열린데이터, Kaggle 등
- 파이프라인: 수집(Airflow) → 적재(S3) → 가공(PySpark / Glue) → 분석(Athena) → 시각화(Tableau / Superset)
- 배포: Terraform IaC + GitHub Actions CI/CD
- 문서화: 아키텍처 다이어그램 + 의사결정 기록(ADR)

> 메시지: "18년 경력자가 최신 스택으로 처음부터 끝까지 설계한 모습"을 증명

### 3순위 (여유 시)

#### 6. 오픈소스 소량 기여

- Airflow, Spark, Databricks 등 사용 중인 툴 저장소에
- 문서 오타 수정, 한국어 번역, 작은 버그 수정 PR 1~2개
- **양보다 "기여 흔적" 자체가 신호**

## 포트폴리오 사이트에 반영할 구조 (권장)

현재 사이트에 다음 섹션을 추가하면 이 약점을 정면으로 보완할 수 있습니다.

### A. "Code & Writings" 섹션 신설 (GitHub 링크보다 상위 개념)

```
┌─────────────────────────────────────────────────┐
│  Code & Writings                                │
│  실무 코드는 기업 NDA로 비공개이며,             │
│  블로그와 샘플 레포로 경험을 공유합니다.        │
├─────────────────────────────────────────────────┤
│  [블로그 글 카드 1]   [블로그 글 카드 2]        │
│  [샘플 레포 카드 1]   [샘플 레포 카드 2]        │
│                                                  │
│  📂 GitHub Profile →                             │
└─────────────────────────────────────────────────┘
```

### B. 프로젝트 카드에 "아키텍처 다이어그램" / "기술 선택 근거" 추가

- 코드가 없어도 **의사결정 과정**이 시니어리티 증명
- SVG로 데이터 흐름 시각화 (3~5 노드 수준이면 충분)
- "왜 Databricks가 아닌 SageMaker를 검토했는가" 수준의 한 줄 근거

## 서류전형 제출 전 최종 체크

- [ ] GitHub 프로필 상단 README에 컨텍스트 배너가 있는가
- [ ] 블로그 글 최소 1편이 공개되어 있는가
- [ ] 대학원 과제 레포에 모두 제대로 된 README가 있는가
- [ ] 포트폴리오 사이트의 "Code & Writings" 섹션에 내용이 들어있는가
- [ ] GitHub 링크를 눌렀을 때 **허전함**이 없는가 (최소 4~6개 레포가 보여야 함)

## 로드맵 (우선순위 + 소요 시간)

| 순서 | 항목 | 예상 소요 | 효과 |
|---|---|---|---|
| 1 | GitHub 프로필 README 배너 | 1시간 | ⭐⭐⭐⭐⭐ |
| 2 | 대학원 레포 README 정리 | 반나절 | ⭐⭐⭐⭐ |
| 3 | 포트폴리오 사이트 "Code & Writings" 섹션 프레임 | 1~2시간 | ⭐⭐⭐⭐ |
| 4 | 기술 블로그 글 1편 | 1~2일 | ⭐⭐⭐⭐⭐ |
| 5 | 샘플 템플릿 레포 1개 | 2~3일 | ⭐⭐⭐ |
| 6 | End-to-End 토이 프로젝트 | 1~2주 | ⭐⭐⭐⭐ |
| 7 | 블로그 글 2~3편 누적 | 지속 | ⭐⭐⭐⭐⭐ |

> **추천 진행 순서**: 1 + 3 동시 진행 → 2 → 4 (블로그 글) → 5 (템플릿) → 6 (토이 프로젝트)

## 핵심 메시지 정리

1. **코드 양보다 사고의 깊이** — 시니어 평가의 본질
2. **블로그가 최고의 대체재** — NDA 안전, 사고력 증명, 검색 노출
3. **있는 것을 잘 포장하기부터** — 대학원 과제 README 정리가 의외로 빠른 개선
4. **채용 담당자에게 먼저 설명하기** — 프로필 배너로 오해 원천 차단
5. **템플릿 레포는 "패턴의 언어"** — 실제 코드를 공개하지 않아도 실력 증명 가능
